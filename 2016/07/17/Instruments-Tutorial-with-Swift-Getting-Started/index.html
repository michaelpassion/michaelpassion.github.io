Instruments Tutorial with Swift: Getting Started

发表于 Jul 17 2016   |   分类于 tutorial   |     |   阅读次数

此教程来自 raywenderlich.com ，介绍了使用 Instruments 的最重要的几个方面。Instruments 可以用来检查代码的性能，内存，引用循环，等其他问题。
通过此教程，你可以了解以下内容：

  1. 使用 Time Profiler 工具来查找代码中的 hot-spots， 从而使你的代码更加高效
  2. 使用 Allocations 工具检测和修复内存管理问题，如代码中得 strong reference cycles

开始

本教程不会从创建 APP 开始，我们将使用一个准备好的 project， 你的任务是快速了解这个 APP 然后跟随教程来优化这个 APP – 和你将来要优化自己的 apps 一样。下载地址

这个 app 使用 Flickr API 来搜索图片。你需要一个API key 来使用这些 API。你可以从http://www.flickr.com/services/api/explore/?method=flickr.photos.search 搜索任何东西，从页面底部生成的 URL 中复制 API key。 API key 在 &api_key= 和 下一个 & 之间的部分。比如如果 URL 是：
http://api.flickr.com/services/rest/?method=flickr.photos.search&
api_key=6593783efea8e7f6dfc6b70bc03d2afb&format=rest&api_sig=f24f4e98063a9b8ecc8b522b238d5e2f
则API key 是： 6593783efea8e7f6dfc6b70bc03d2afb.
将API key 粘贴到 FlickrSearcher.swift 文件顶部的相应位置，替换文件中得 API key。
需要注意的时，这个 API key隔几天会变动，所以你需要时不时地重新生成key。App 会在当前的key失效后提示你。
build and run, 随便搜索下，点击搜索结果，会有以下以下效果：

本教程接下来的部分会展示怎么发现并修改这个app中得问题。你也将见识到 instruments 会将 debugging 问题变得多么容易 :]

Time for Profiling

第一个学习的工具是 Time Profiler, Instruments 会在一定的时间段中止程序的执行并记录每一个运行中的线程的 stack trace栈轨迹。你可以把这看做按下了Xcode dubugger 中的暂停键。
先看一下 Time Profiler 的样子。

这张图片展示是 Call Tree。Call Tree 展示了app中多种方法的运行时间。每一行都是一个不同的方法，在其后是程序的执行路径。每个方法执行的时间可以通过 prifiler停留在每个方法中的次数来计算。
例如， 如果在100个1毫秒的样本中，某个特定的方法出现在10个样本中的栈的栈顶，那么可以推断出将近10%的时间 – 10毫秒 –在执行这个方法。这种估计方式也许粗暴，但是能用！
注意：大多数情况下，你应该在真实设备上 profile 你的 app，而不是模拟器上。iOS模拟器后是你MAC电脑的所有资源，而一个iOS设备存在各种手机硬件的限制。也许你的app再模拟器中运行的很好，但是运行在只能是设备中就会发现各种运行问题。
闲话少说，开始 instrumenting!
从Xcode的菜单栏中，选择 Product\Profile, 或快捷键 ⌘I。这样会创建这个 app 并运行 Instruments. 你会看到这样的可选择的窗口：

这些是 Instruments 各种模板。

在 Instruments 窗口中可以看到时间在不停增长，一个小箭头从屏幕中央的图表中自左向右移动。这表示这个 app 正在运行。

深入研究

执行一次图片搜索，并查看搜索结果。我个人喜欢搜索“小狗”，但是尽管搜索你想搜索的-万一你是一个猫粉呢 :]
现在，将图片列表上下滑动几次以便在Time Profiler中得到多一些的数据。你应该会注意到敞口中间的数字在不断变化而且图表在不断写入数据；这意味着cpu周期在被使用。
你可能没想过有UI能卡城这样；只有列表滑动起来如黄油一般丝滑才可以真正用来展示！为了丁伟问题，你需要设置一些选项。
在右手边，选择 Display Settings 面板(快捷键 ⌘+2)。在面板中 Call Tree部分，选择 Separate by Thread, Invert Call Tree, Hide Missing Symbols, Hide System Libraries。如下图所示：

下面是每个选项对左手边所展示数据起到的作用：

  • Separate by Thread 每个线程要单独考虑。通过这个选项可以知道那个线程使用的CPU最多。
  • Invert Call Tree 选中这个选项，栈轨迹从顶向低展示。通常这正是你想要的，因为你想看到的是占用CPU时间最多的最底层的方法。
  • Hide Missing Symbols 如果找不到你应用或是系统框架的 dSYM 文件，那么你将只能看到二进制中地址对应的16进制的值，而不是方法名（符号）。选中这个选项，则只有完全解析的符号会展示出来，没有解析的16进制的值将被隐藏。这有助于整理当前展示的数据。
  • Hide System Libraries 这个选项被选中时，只显示你自己 app 中的方法。通常情况下会选中这个选项，因为你只关心那些CPU时间花费在你的代码中 – 你对系统软件使用的CPU时间也没啥办法！
  • Flatten Recursion 这个选项将每个栈轨迹中的递归函数(自己调用自己)作为一个入口来处理，而不是多个。
  • Top Functions 选中这个选项，Instruments会将花费在方法自身花费的时间和在方法中调用其他方法，其他方法花费的时间一起作为该方法的总时间。即如果方法A中调用了方法B，则A所花费的时间为在A消耗的时间加上方法B的时间。这个机制很有用，它使你在深入调用栈时可以选出耗时最多的而一个，从而减少最耗时的方法的时间。
  • 如果你在运行一个 Objective-C 应用，还有一个选项是Show Obj-C Only： 如果选中了，则只有Objecti-C方法会展示，而不是所有的C或C++方法。可能你的程序中没有C或C++方法，但是如果你在调优的是一个 OpenGL 应用，就可能会有一些C++函数。

虽然一些值可能不太一样，如果你选中了以上选项，则各个入口的顺序应该和以下表格差不多。
